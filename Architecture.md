# Архитектура сервиса прогнозирования дефолта

## Монолит vs микросервисы
Сервис реализован в **монолитном** стиле: препроцессинг, модель и веб-интерфейс (Flask) упакованы в один Docker-образ.  
**Обоснование:**
- один эндпоинт (`/predict`) и небольшой объём кода не требуют микросервисного разделения.
- простота развёртывания и отладки (один контейнер вместо нескольких).
- отсутствие необходимости независимого масштабирования компонентов на данном этапе.
- при росте нагрузки монолит можно разделить: вынести пайплайн предобработки в отдельный сервис, а модель - за балансировщик с версионированием.

## Поддержка нескольких версий модели
Внутри сервиса загружаются две версии модели (v1 и v2) на основе RandomForest с разными гиперпараметрами.  
Клиент может указать желаемую версию в запросе (`"version": "v2"`). Если версия не указана, сервис случайным образом (50/50) выбирает одну из загруженных моделей. Это автоматически реализует разделение трафика для A/B-тестирования без развёртывания отдельных экземпляров сервиса или внешнего маршрутизатора.

## Брокер сообщений (RabbitMQ) - концепт
При масштабировании на пакетную обработку или при высокой нагрузке можно внедрить **RabbitMQ**:
- клиент отправляет батч-запрос в очередь `predict_requests`.
- контейнеры с моделью разбирают очередь, выполняют предсказания и складывают результаты в очередь `predict_results`.
- повышается отказоустойчивость: если воркер упал, сообщение вернётся в очередь.
- возможна асинхронная запись логов и метрик в отдельную очередь для мониторинга.

## Логирование и мониторинг
- все запросы и ответы логируются в формате **JSON** с помощью библиотеки `python-json-logger`.
- логи выводятся в stdout контейнера и дублируются в терминал сервера.
- в production-среде их можно собирать с помощью **ELK-стека** (Filebeat - Elasticsearch - Kibana) для построения дашбордов по количеству запросов, распределению вероятностей дефолта и ошибкам.
- в Docker Compose добавлен **Nginx** в качестве обратного прокси; его access-логи (`nginx/logs/`) эмулируют сбор логов с фронтального сервера.

## Оптимизация модели: ONNX-ML
Для ускорения инференса и развёртывания вне Python можно конвертировать обученный RandomForest в формат **ONNX**:
python

from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
initial_type = [('float_input', FloatTensorType([None, 13]))]
onnx_model = convert_sklearn(model, initial_types=initial_type)
with open("model.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())

ONNX-модель можно запускать в ONNX Runtime, что снижает зависимость от Python.    

## uWSGI + NGINX в production
В промышленной среде встроенный сервер Flask заменяют связкой:
uWSGI - application-сервер, управляющий процессами Python, обеспечивающий многопоточность и асинхронную обработку.

NGINX - обратный прокси, принимающий HTTP-запросы от клиентов, раздающий статику, обрабатывающий медленные соединения и проксирующий динамические запросы к uWSGI.
Это повышает производительность, безопасность и позволяет обслуживать сотни одновременных запросов.

## Инструменты MLOps (обзорно):

Полный цикл модели включает: сбор данных - EDA - предобработка - обучение - оценка - сохранение - развёртывание - мониторинг - A/B-тестирование - переобучение.

### DVC (Data Version Control)
- **Версионирование данных** - фиксирует версию датасета `UCI_Credit_Card.csv` и обработанных файлов (`X_train.npy`, `y_train.npy`). Любое изменение данных сразу видно в истории.
- **Воспроизводимость** - команда `dvc repro` автоматически перезапускает пайплайн (`preprocess.py - train.py - train_v2.py`), если изменились исходные данные или код. Это гарантирует, что обучение всегда можно повторить с теми же результатами.
- **Хранение артефактов** - крупные файлы (модели, обработанные данные) хранятся в удалённом хранилище (S3, GCS), а в Git остаются только лёгкие ссылки. 

### MLflow Tracking
- **Отслеживание экспериментов** - при каждом запуске `train.py` или `train_v2.py` MLflow автоматически логирует:
  - **Параметры** (n_estimators, max_depth, class_weight и др.)
  - **Метрики** (F1-score, Precision, Recall, ROC-AUC)
  - **Артефакты** (путь к сохранённой модели, confusion_matrix)
  - **Сравнение версий** - в MLflow UI можно на одном экране сравнить запуски v1 и v2 по всем метрикам, выбрать лучшую модель и зарегистрировать её в Model Registry.
  - **Model Registry** - после выбора лучшей версии модель можно пометить как `Staging` или `Production`, что напрямую связано с A/B-тестированием и последующим развёртыванием.
