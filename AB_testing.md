# План A/B-тестирования модели прогнозирования дефолта

## Цель
Сравнить текущую модель (v1) с новой версией (v2) по качеству предсказания дефолта и влиянию на бизнес-метрики.

## Дизайн эксперимента
- **Участники**: клиенты, обращающиеся к сервису за предсказанием дефолта.
- **Группы**:
  - **Контрольная группа (A)**: клиенты, для которых предсказание выполняется текущей моделью (v1).
  - **Тестовая группа (B)**: клиенты, для которых предсказание выполняется новой моделью (v2).
- **Способ разделения**: каждый запрос случайным образом направляется к v1 или v2 с равной вероятностью 50/50. Распределение происходит на стороне сервера до вызова модели.

*Примечание*: в условиях дисбаланса классов (22% дефолтов) при большом количестве наблюдений (более 10 000) случайное разделение 50/50 на уровне запроса обеспечивает статистическую эквивалентность групп. Для дополнительного контроля на этапе анализа можно проверить баланс классов в группах; если обнаружено значимое различие, применить пост‑стратификацию или увеличить продолжительность теста.

## Метрики оценки
### Основная метрика
- **F1-score для класса дефолта** - гармоническое среднее между точностью (Precision) и полнотой (Recall). Выбрана как основная, потому что дисбаланс классов (22% дефолтов) делает Accuracy неинформативной, а F1-score одинаково учитывает оба типа ошибок.

### Дополнительная метрика
- **Recall (полнота)** - доля истинных дефолтов, которые модель смогла обнаружить. Выбрана, так как пропуск дефолта обходится банку дороже, чем ложная тревога

### Бизнес-метрика
- **Ожидаемые финансовые потери (Expected Loss)** - сумма произведений `вероятности дефолта` и `суммы задолженности` для False Negative предсказаний. Позволяет оценить, какая из моделей минимизирует потенциальные убытки.

### Расчёт размера выборки, продолжительности теста и анализ результата теста

**Планирование размера выборки**
- Для определения минимального количества дефолтов в каждой группе используется метрика **Recall**, несмотря на то что основной метрикой является F1‑score. Причина: для F1‑score нет простой формулы расчёта выборки, так как она является нелинейной комбинацией Precision и Recall. Для Recall как пропорции существует готовая формула Z‑теста, которая даёт консервативную (верхнюю) оценку необходимого объёма данных,тем самым обеспечивая мощность и для F1.
- **Ожидаемый baseline (Recall v1)**: 0.59.
- **Минимальная детектируемая разница (MDE)**: 0.04 (4 п.п.). Изменение такого размера будем считать бизнес‑значимым.
- **Уровень значимости (α)**: 0.05.
- **Мощность (1−β)**: 0.80.
- **Расчёт по формуле Z‑теста** даёт примерно 2060 дефолтов в каждой группе.
- **Общее количество запросов** (при доле дефолтов 22%):примерно 9364 запросов на группу, всего около **18 700 запросов**.
- **Оценка продолжительности**: при среднем потоке ок. 1500 запросов в день потребуется ок.**2 недель** (считаем с запасом - ориентир более 10 000 запросов на группу).

*Примечание*: при большом числе наблюдений случайное разделение 50/50 на уровне запроса обеспечивает статистическую эквивалентность групп. На этапе анализа проверяется баланс классов; при значимом дисбалансе применяется пост‑стратификация или тест продлевается.


**Метод статистического анализа**
- **Основная метрика**: F1‑score для класса дефолта.
- **Дополнительная метрика** Recall. Выбор данной метрики обусловлен тем, что банку дороже обходятся пропущенные дефолты, чем излишняя осторожность.
- **Статистический тест**: Z‑тест для сравнения пропорций. При больших выборках (>2000 дефолтов) распределение выборочного F1‑score хорошо аппроксимируется нормальным, что делает Z‑тест корректным.
- **Нулевая гипотеза H₀**: F1‑score(v2) − F1‑score(v1) = 0.
- **Альтернатива H₁**: ≠ 0 (двусторонний тест).
- **P‑value**: вероятность получить наблюдаемую (или бо́льшую) разницу при условии, что H₀ верна. Если **p‑value < 0.05**, различие признаётся статистически значимым.
- **95% доверительный интервал (CI)** для разницы F1‑score строится методом **бутстрапа** (1000 ресемплов). Если CI не содержит 0, это подтверждает статистическую значимость.

**Критерий успешности**
- **Статистическая значимость**: p‑value < 0.05 (или 95% CI для разницы не включает 0).
- **Практическая значимость**: наблюдаемая разница F1‑score больше или равно MDE (0.04). Только сочетание статистической и практической значимости позволяет рекомендовать v2 к внедрению.
- **Итоговое решение**:
  - Если p‑value < 0.05 **и** разница более и равна 0.04, v2 признаётся значимо лучшей и рекомендуется к замене.
  - Если p‑value более 0.05 или разница < 0.04, текущая версия v1 сохраняется.
**Дополнительная метрика** Recall: значимое улучшение при стабильном Precision укрепляет решение о внедрении v2.
- **Бизнес‑метрика Expected Loss** является финальным фильтром: v2 не должна увеличивать ожидаемые финансовые потери.

## Влияние A/B-тестирования на архитектуру сервиса
Для поддержки A/B-тестов в архитектуру сервиса были внесены следующие изменения:
- **Загрузка нескольких версий модели** - при старте сервис загружает модели v1 и v2 в словарь `models`, что позволяет динамически выбирать версию без перезапуска.
- **Параметр `version` в API** - эндпоинт `/predict` принимает опциональное поле `version`. Если оно не указано, сервер случайным образом (с вероятностью 50/50) выбирает одну из загруженных моделей (v1 или v2), что автоматически обеспечивает разделение трафика для A/B-теста.
- **Логирование версии** - каждый запрос и ответ логируются с указанием использованной версии модели (`model_version`), что обеспечивает сбор данных для последующего анализа результатов теста.
- **Независимость моделей** - модели v1 и v2 используют один и тот же пайплайн предобработки (`full_pipeline.joblib`), но имеют разные артефакты (`model.joblib` и `model_2.joblib`), что упрощает добавление новых версий в будущем.
